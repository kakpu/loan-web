# CLAUDE.md — Claude Code プロジェクトルール

このファイルは Claude Code に読み込ませるプロジェクト固有のルール集です。
すべての作業においてこのドキュメントを最優先で参照してください。

---

## 1. 基本方針

- **回答言語**: すべての回答・コメント・説明は **日本語** で行う。
- **コードコメント**: 日本語を使用する。英語のコメントを日本語に修正する場合も同様。
- **PoC優先**: このプロジェクトは1人月のPoC（概念実証）。過剰な設計・実装より **動くデモを最優先** にする。
- **ドキュメント参照**: 設計変更・実装前に必ず `docs/` 配下の設計書を参照すること。

---

## 2. プロジェクト概要

- **システム名**: Webローン申込システム（PoC）
- **チャネル**: LINE Webhook 起点
- **決済**: PayPay のみ実装（他はUI表示のみ）
- **外部API**: すべてダミー応答（即時与信・eKYC・反社チェック・PayPay）
- **設計書の場所**:
  - `docs/01_requirements.md` — 要件定義・業務状態一覧・状態遷移
  - `docs/02_architecture.md` — アーキテクチャ設計
  - `docs/03_database.md` — データベース設計
  - `docs/04_api.md` — API設計
  - `docs/05_sitemap.md` — サイトマップ・画面設計

## 3. 業務ルール（変更禁止）

以下の業務状態・遷移は **移行要件** であり、実装で変えてはならない。

### 業務状態一覧

| state ID | 状態名 | 説明 |
|---|---|---|
| S01 | 申込開始 | Web申込入力中 |
| S02 | 申込受付済 | 申込情報が確定 |
| S03 | 即時与信済 | 借入可能額を提示済 |
| S04 | 本人確認待 | 本人確認未完了 |
| S05 | 審査中 | マニュアル審査中 |
| S06 | 契約可 | 契約書署名待ち |
| S07 | 会員確定 | 契約完了 |
| S99 | 断り確定 | 審査否決 |

### 状態遷移の鉄則

- 状態を新規追加・既存状態を削除する場合は必ず要件定義書を確認して合意を得ること。
- 遷移の起点・終点を誤実装しない（例：S03 → S05 は存在しない）。
- 審査否決理由は **顧客に非開示**（UI・APIレスポンスに含めない）。

## 4. コミットメッセージ規約

### フォーマット

```
<type>(<scope>): <概要（日本語）>

<本文（任意・日本語）>
```

### type 一覧

| type | 用途 |
|---|---|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `refactor` | 動作変更を伴わないリファクタリング |
| `style` | フォーマット・スタイルの修正（ロジック変更なし） |
| `docs` | ドキュメントのみの変更 |
| `test` | テストコードの追加・修正 |
| `chore` | ビルド設定・依存関係などの雑務 |
| `wip` | 作業途中（マージ前に別 type に変更する） |

### scope 例

`application`, `identity`, `review`, `contract`, `payment`, `admin`, `auth`, `db`, `api`

### 記述例

```
feat(application): 申込確定APIとS02遷移を実装

POST /applications/{id}/validate を呼び出し、
バリデーション通過後にステータスをS02へ更新する。
```

```
fix(review): 審査否決時にS99へ遷移しない不具合を修正
```

### 禁止事項

- 英語の概要文（type と scope は英語でよい）
- `fix: 修正` のような情報量ゼロのメッセージ
- 複数の無関係な変更を1コミットにまとめること

## 5. コードスタイルルール

### 5.1 全般

- **Lint必須**: コミット前に `npm run lint` と `npm run typecheck` をパスすること。
- **strict モード**: TypeScript の `strict: true` が有効。`any` 型は原則禁止。やむを得ない場合は `// TODO: 型を適切に定義する` コメントを添える。
- **未使用変数禁止**: `noUnusedLocals` / `noUnusedParameters` が有効。使わない変数は削除する。
- **インデント**: スペース2つ（既存ファイルのスタイルに合わせる）。

### 5.2 React / TypeScript

- **コンポーネント**: 関数コンポーネントのみ使用（クラスコンポーネント禁止）。
- **型定義**: `interface` より `type` を優先する（ただし既存コードの慣習に合わせる）。
- **Props の型**: `React.FC<Props>` より `(props: Props) => JSX.Element` 形式を優先。
- **Hooks ルール**: `eslint-plugin-react-hooks` の推奨ルールに従う。
- **状態管理**: PoC規模のため useState / useReducer で完結させる。外部状態管理ライブラリは原則追加しない。

### 5.3 Tailwind CSS

- クラスの並び順: レイアウト → サイズ → 色 → その他（可読性優先）。
- 動的クラスは `cn()` や `clsx()` を使用し、テンプレートリテラルで文字列結合しない。

### 5.4 Supabase

- クライアントは `src/lib/supabase.ts` などに集約し、直接 `createClient` を各コンポーネントで呼ばない。
- RLS（Row Level Security）ポリシーを意識した実装をする。
- SQL を直接埋め込む場合はパラメータバインドを使い、文字列結合によるクエリを作らない。

## 6. コメントの書き方

### 書くべきコメント

- **なぜそうするのか**（Why）が自明でない処理。
- **業務ルールの参照**（例：「S02→S03遷移条件：メール・携帯・生年月日確定が必要」）。
- **PoC用のダミー処理**には必ず `// [PoC] ダミー応答: 本番実装時は外部API呼び出しに置き換える` を付ける。
- **未実装表示**には `// [PoC] 未実装: 本番では○○が必要` を付ける。

### 書かなくてよいコメント

- コードを読めば明らかな内容（`// ループする` など）。
- 関数名・変数名で意図が伝わるもの。

## 7. Claudeが意識すべき観点

### 7.1 過剰実装の回避（PoC原則）

- PoCで「作らない/書かない」と明記されているものは実装しない。
  - 書類アップロード/書類管理の本実装
  - セキュリティ詳細・監査ログの網羅
  - 負荷対策・スケール設計
- 「将来のために汎用化」「念のためエラーハンドリングを追加」など、指示のない拡張を自発的に行わない。
- 3つ以上の似たコードが出たとき初めて抽象化を検討する。

### 7.2 セキュリティ（PoC範囲内でも守ること）

- **個人情報を含む値**（氏名・メールアドレス・生年月日・携帯番号）をコンソールに出力しない。
- **API キー・シークレット**をコードにハードコードしない。必ず環境変数（`.env`）を使用する。
- XSS 対策: ユーザー入力を `dangerouslySetInnerHTML` に渡さない。
- SQL インジェクション対策: 文字列結合でクエリを組み立てない。
- IDOR 対策: リソースへのアクセス時は必ずログインユーザーの所有確認をする。

### 7.3 業務状態の整合性

- 状態遷移を実装する際は `docs/01_requirements.md` の状態遷移定義を必ず照合する。
- 想定外の状態への遷移（例：S07 → S01 への戻り）は実装しない。
- 状態変更は「状態変更のみを担うAPI/関数」に集約し、UIコンポーネント内で直接 DB を更新しない。

### 7.4 冗長なコードの回避

- 同じロジックを複数箇所にコピーしない。共通処理は `src/lib/` や `src/utils/` に切り出す。
- 使わなくなったコードは削除する（コメントアウトして残さない）。
- 関数は単一責任の原則に従い、1関数が1つのことをするようにする。

### 7.5 ファイル変更前の確認

- 既存ファイルを編集する前に必ず `Read` ツールで内容を読む。
- 影響範囲が広いリファクタリングは、着手前にユーザーに確認する。

## 9. 開発コマンド

```bash
npm run dev        # 開発サーバー起動
npm run build      # 本番ビルド
npm run lint       # ESLint 実行
npm run typecheck  # TypeScript 型チェック
npm run preview    # ビルド結果のプレビュー
```

コミット前に **lint と typecheck の両方がエラーなしであること** を確認する。
