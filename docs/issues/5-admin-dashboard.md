### 背景 / 目的
`AdminDashboard.tsx` が申込一覧・詳細・審査入力を 1 ファイルに持ち、`alert()` でフィードバックを表示している。サイトマップ設計（`05_sitemap.md`）に従いページを分割し、審査フローを API 設計（`04_api.md`）の API-07・08 相当のフローに整合させる。

- 依存: #1, #4
- ラベル: frontend

### スコープ / 作業項目
- `/admin`（申込一覧）: 状態フィルタ・一覧テーブルの改善
- `/admin/applications/:id`（申込詳細）: 申込情報・与信結果・遷移ログの表示
- 審査入力フォーム（本人確認補完 OK/NG・在籍確認 OK/NG・可決/否決）→ `review_results` INSERT → `state` 更新
- タイムアウト擬似発火ボタン（S04 の申込のみ表示）→ S99 遷移
- `alert()` を廃止してインラインフィードバックに置き換え
- `AdminDashboard.tsx` の解体と不要コードの削除

### ゴール / 完了条件（Acceptance Criteria）
- [ ] `/admin` に申込一覧が表示され、状態バッジ（色分け）と状態フィルタ（ドロップダウン）が動作する
- [ ] `/admin/applications/:id` に申込情報・与信額・状態遷移ログが表示される
- [ ] S05 の申込に対して本人確認補完・在籍確認・可決/否決の審査入力フォームが表示される
- [ ] 審査結果登録後、`review_results` テーブルに INSERT され、`state` が S06（可決）or S99（否決）に更新される
- [ ] S04 の申込にタイムアウト擬似発火ボタンが表示され、押下で `state` が S99 に遷移する
- [ ] `alert()` の呼び出しがコンポーネントから除去され、インライン通知（成功バナー or エラーバナー）に置き換わっている

### テスト観点
- 手動: S05 の申込を選択し、可決・否決それぞれの審査を実施して状態遷移を確認
- 手動: S04 の申込でタイムアウトボタンを押し S99 へ遷移することを確認
- 手動: 状態フィルタで S05 のみ表示されることを確認
- 検証方法: Supabase ダッシュボードで `review_results` と `state_transitions` テーブルのレコードを目視確認
