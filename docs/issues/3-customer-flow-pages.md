### 背景 / 目的
`CustomerFlow.tsx` がすべての申込ステップを 1 ファイル・`useState` で管理しており、URL と業務状態が一致しない。サイトマップ設計（`05_sitemap.md`）に従い状態別のページコンポーネントへ分割し、URL を業務状態の正規表現として機能させる。

- 依存: #1, #2
- ラベル: frontend

### スコープ / 作業項目
- `/apply/:id` アクセス時に DB の `state` を読み取り対応ページへリダイレクト
- 状態別ページの実装（S03・S04・S05・S06・S07・S99）
- S05 の審査待ち画面（現行 `CustomerFlow.tsx` に存在しない）を新規追加
- 否決理由を画面・API レスポンスに含めない制御（要件定義準拠）
- `CustomerFlow.tsx` の解体と不要コードの削除

### ゴール / 完了条件（Acceptance Criteria）
- [ ] `/apply/:id` へアクセスすると DB の `state` を読んで対応する URL へ自動リダイレクトされる
- [ ] `/apply/:id/credit`（S03）— 与信可: 仮与信額表示と「次へ進む」ボタン、与信否: `/apply/:id/rejected` へ自動遷移
- [ ] `/apply/:id/identity`（S04）— 本人確認案内と「書類アップロード（未実装）」表示が実装されている
- [ ] `/apply/:id/waiting`（S05）— 審査中メッセージと `GET /status` ポーリング（30 秒間隔）が実装されている
- [ ] `/apply/:id/contract`（S06）— 契約書 PDF リンクと「同意して署名する」ボタンが実装されている
- [ ] `/apply/:id/complete`（S07）— 決済完了メッセージが表示される
- [ ] `/apply/:id/rejected`（S99）— 否決理由を含まない断りメッセージが表示される

### テスト観点
- 手動: Supabase ダッシュボードで `state` を各値に書き換え、`/apply/:id` にアクセスして正しいページへ遷移することを確認
- 手動: S99 ページで否決理由のテキストが含まれないことを目視確認
- 検証方法: ブラウザの URL バーと表示内容で状態・画面の対応を確認

要確認事項:
- S05 のポーリング間隔（30 秒で良いか、デモ用に短縮するか）
